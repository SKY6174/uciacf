# 위원회 운영 시스템 계획서

> 버전: 1.0.0 | 작성일: 2026-07-23 | 상태: Approved for development/staging implementation
> 프로젝트 수준: Dynamic (Next.js + Supabase + Vercel)

## 1. 목적

산학협력단 대시보드 안에서 위원회 개설부터 위원별 자료 열람, 의안 심의, 서명, 참여현황, 종합분석, 결과보고서 보관까지 하나의 감사 가능한 흐름으로 처리한다. PDF 원문은 Supabase Storage의 비공개 `meeting_docs` 버킷에 저장하고 권한 확인 후 단기 signed URL로만 열람한다.

## 2. 사용자와 핵심 과업

| 사용자 | 핵심 과업 |
|---|---|
| 관리자/간사 | 위원회·위원·의안 생성, PDF 업로드, 보안코드 발급, 참여 모니터링, 결과보고서 생성 |
| 위원장/위원 | 보안코드 로그인, 의안별 PDF 열람, 찬성·반대·기권 및 의견 제출, 전자서명 |
| 감사자/책임자 | 제출 이력, 문서 접근, 결과 집계, 서명 증적과 보고서 조회 |

## 3. 범위

### 3.1 이번 구현(MVP)

- 위원회 종류·상태·일정과 위원장/위원/간사 구성
- 위원별 일회성 초기 접근코드 설정 및 해시 보관
- 의안과 PDF 전용 첨부파일 메타데이터
- `meeting_docs` 비공개 버킷, RLS/Storage 정책, signed URL 기반 인앱 PDF 열람
- 의안별 심의(찬성/반대/기권, 의견), 제출 잠금과 감사로그
- 위원 참여현황과 의안별 집계
- 근거가 표시되는 AI 분석 저장 구조 및 안전한 기본 요약
- 이름 입력+동의 체크 기반 전자서명 증적과 결과보고서 PDF 생성 API
- 관리자용 위원회 워크스페이스와 위원 전용 로그인/심의 페이지

### 3.2 제외 및 후속 범위

- 공동인증서·전자서명인증사업자 기반 법적 공인전자서명
- 이메일/SMS 보안코드 자동 발송, SSO, MFA
- 실제 운영 개인정보와 운영 Supabase 연결
- 생성형 AI 공급자·모델 확정 전 운영 의사결정 자동화
- PDF 원문 OCR, 주석 협업, 워터마크/DRM

## 4. 핵심 정책

1. 접근코드는 평문 저장·조회하지 않고 `pgcrypto.crypt` 해시로만 비교한다.
2. 위원 세션 토큰도 해시 보관하며 HttpOnly, Secure, SameSite=Lax 쿠키를 사용한다.
3. 원문은 public URL을 만들지 않고, 서버가 위원회 권한을 재확인한 후 5분 signed URL을 발급한다.
4. MIME, 확장자, 최대 크기를 서버와 Storage 양쪽에서 검증한다. 허용 형식은 PDF뿐이다.
5. 심의 제출 이후 수정은 관리자 재개방과 감사로그 없이는 허용하지 않는다.
6. 전자서명은 서명자 이름, 동의문 버전, 시각, IP 해시, User-Agent 해시, 심의 스냅샷을 보존하는 일반 전자서명 증적이다.
7. AI 분석은 원문·개인정보 전체를 무조건 외부로 보내지 않는다. 집계와 비식별 의견을 입력으로 하고 결과·모델·프롬프트 버전·근거를 기록한다.
8. 운영데이터 투입은 기존 상세 설계의 구현 착수 게이트 승인 후에만 진행한다.

## 5. 성공 기준

- [ ] 허용된 위원만 자신의 위원회와 PDF signed URL을 얻는다.
- [ ] PDF가 아닌 파일과 위장 MIME/확장자 업로드가 거절된다.
- [ ] 위원은 모든 의안을 확인하고 심의를 제출한 뒤 서명할 수 있다.
- [ ] 간사는 미접속/열람/심의/서명 상태를 구분해 확인한다.
- [ ] 결과 집계와 보고서 수치가 제출 행까지 추적된다.
- [ ] 관리자와 위원의 모든 민감 변경이 append-only 감사로그에 남는다.
- [ ] build, typecheck, lint와 핵심 RLS 검증이 통과한다.

## 6. 위험과 대응

| 위험 | 대응 |
|---|---|
| 공유 접근코드 유출 | 위원별 코드, 시도 제한, 만료/비활성화, 해시 저장, 세션 만료 |
| Storage URL 유출 | 비공개 버킷, 짧은 signed URL, 서버 권한 재검증 |
| 서명의 법적 효력 오인 | UI·보고서에 일반 전자서명임을 명시하고 공인전자서명은 후속 연계 |
| 악성 PDF | MIME/확장자/크기 검증, 다운로드 강제 방지, 향후 악성코드 스캔 게이트 |
| AI 환각·민감정보 유출 | 근거 수치 표시, 비식별 집계만 전송, 사람 검토/승인 상태 |
| 운영 정책 미승인 | 개발/스테이징 합성 데이터만 사용하고 운영 연결 차단 |

## 7. 구현 단계

4. 관리자/위원 UI와 PDF 리더 구현
5. 결과보고서 생성·렌더링 검증
6. Gap 분석과 운영 전 승인항목 정리
